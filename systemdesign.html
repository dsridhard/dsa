System Design Key Concepts

Scalability

Concept: The ability of a system to handle a growing amount of work by adding resources.

Details:

  • Vertical Scaling (Scaling Up): Increasing the resources of a single server (e.g., more CPU, RAM).
  • Horizontal Scaling (Scaling Out): Adding more servers to the system to distribute the load.

Important Questions:

  • How would you scale this system to handle 10x the traffic?
  • What are the trade-offs between vertical and horizontal scaling for this application?
  • How can we design the system to be stateless to make horizontal scaling easier?

Availability

Concept: The measure of a system's uptime and accessibility to users.

Details:

  • Expressed as a percentage (e.g., 99.999% "five nines").
  • Achieved through redundancy, failover mechanisms, and eliminating single points of failure.

Important Questions:

  • What are the single points of failure in this design?
  • How can we ensure high availability for the database?
  • What happens if a server or a data center goes down?

Reliability

Concept: The probability that a system will produce correct outputs and not fail.

Details:

  • Closely related to availability but focuses on data correctness and fault tolerance.
  • Involves techniques like data replication and checksums.

Important Questions:

  • How do we prevent data corruption?
  • How can the system recover from a partial failure?
  • What guarantees does the system provide regarding data integrity?

Performance

Concept: The speed and responsiveness of a system.

Details:

  • Measured by latency (delay) and throughput (operations per second).
  • Improved by optimizing code, using caching, and efficient database queries.

Important Questions:

  • What are the key performance metrics for this system?
  • How can we reduce the latency for end-users?
  • Where are the potential bottlenecks in this design?

Consistency

Concept: Ensuring that all nodes in a distributed system see the same data at the same time.

Details:

  • Strong Consistency: All reads get the most recent write.
  • Eventual Consistency: Over time, all nodes will become consistent.

Important Questions:

  • What level of consistency is required for this application?
  • What are the trade-offs of choosing strong vs. eventual consistency?
  • How does the choice of database affect consistency?

Load Balancing

Concept: Distributing incoming network traffic across multiple servers.

Details:

  • Algorithms: Round Robin, Least Connections, IP Hash.
  • Prevents any single server from becoming a bottleneck.

Important Questions:

  • What load balancing strategy would you use and why?
  • How does the load balancer handle server failures?
  • Where would you place the load balancer in the architecture?

Caching

Concept: Storing frequently accessed data in a temporary, fast-access layer.

Details:

  • Reduces latency and load on the backend systems (like databases).
  • Cache-aside, Read-through, Write-through are common caching patterns.
  • Cache invalidation is a critical challenge.

Important Questions:

  • What data should be cached?
  • What is the cache eviction policy?
  • How do you handle cache invalidation to ensure data consistency?

Database Scaling

Concept: Techniques to handle a growing amount of data and database traffic.

Details:

  • Replication: Creating copies of the database (Master-Slave, Master-Master). Good for read scaling.
  • Sharding (Partitioning): Splitting data across multiple databases. Good for write scaling.

Important Questions:

  • How would you scale the database for a read-heavy workload?
  • When would you consider sharding the database? What is the sharding key?
  • What are the challenges of database replication and sharding?

CAP Theorem

Concept: A distributed system can only provide two of the following three guarantees: Consistency, Availability, and Partition Tolerance.

Details:

  • In modern distributed systems, partition tolerance is a must.
  • The trade-off is typically between Consistency and Availability (CP vs. AP systems).

Important Questions:

  • Does this system need to be more consistent or more available?
  • How does the CAP theorem influence your choice of database or data store?
  • Can you give an example of a CP system and an AP system?

Microservices vs. Monolithic Architecture

Concept: Two different architectural styles for building applications.

Details:

  • Monolith: The entire application is built as a single, unified unit.
  • Microservices: The application is broken down into a collection of smaller, independent services.

Important Questions:

  • What are the pros and cons of using microservices for this project?
  • How do services communicate with each other in a microservices architecture?
  • How do you handle deployment and monitoring in a microservices environment?

Message Queues

Concept: A form of asynchronous communication between services.

Details:

  • A producer sends a message to a queue, and a consumer retrieves it.
  • Decouples services and helps with load leveling and background processing.
  • Examples: RabbitMQ, Kafka, AWS SQS.

Important Questions:

  • Where can we use a message queue in this system to improve performance or reliability?
  • What happens if a message fails to be processed?
  • What are the trade-offs between different message queue technologies?

Content Delivery Network (CDN)

Concept: A geographically distributed network of servers that cache content closer to end-users.

Details:

  • Reduces latency for serving static assets like images, videos, CSS, and JavaScript.
  • Reduces the load on your origin servers.

Important Questions:

  • What kind of content would you serve from a CDN?
  • How does a CDN improve user-perceived performance?
  • What is the strategy for updating content in the CDN?

Advanced System Design Topics

SQL vs. NoSQL

Concept: Two major categories of databases.

Details:

  • SQL (Relational): Structured data, schema-on-write, guarantees ACID properties. Examples: MySQL, PostgreSQL.
  • NoSQL (Non-relational): Unstructured or semi-structured data, schema-on-read. Categories include Key-Value, Document, Column-family, and Graph. Examples: MongoDB, Cassandra, Redis.

Important Questions:

  • What are the data and query patterns for this application?
  • Does the system require strong transactional guarantees?
  • How will the database scale as data volume and traffic grow?

Consistent Hashing

Concept: A hashing technique that minimizes the number of keys that need to be remapped when a cache server is added or removed.

Details:

  • Maps servers and keys to a logical ring. A key is stored on the server that appears first in clockwise order.
  • Crucial for distributed caching and database sharding to reduce churn.

Important Questions:

  • How can we minimize data redistribution when scaling our cache cluster?
  • What are the potential issues with a naive hashing approach for data distribution?
  • How does consistent hashing help in building a scalable distributed system?

Rate Limiting

Concept: Controlling the rate of traffic sent or received by a network interface controller.

Details:

  • Protects services from being overwhelmed by too many requests, preventing DoS attacks and abuse.
  • Algorithms: Token Bucket, Leaky Bucket, Fixed Window, Sliding Window Log.

Important Questions:

  • How would you design a rate limiter to protect our API from abuse?
  • Should the rate limiter be implemented on the client-side, server-side, or at the API gateway?
  • How do you handle rate limiting in a distributed environment?

API Design: REST vs. GraphQL vs. gRPC

Concept: Different paradigms for building APIs.

Details:

  • REST: Resource-based, uses standard HTTP methods, stateless.
  • GraphQL: A query language for APIs, allows clients to request exactly the data they need.
  • gRPC: High-performance RPC framework, uses Protocol Buffers for serialization, works over HTTP/2.

Important Questions:

  • What are the pros and cons of each paradigm for our use case?
  • Is the API for internal service-to-service communication or for a public-facing client?
  • How can we handle API versioning?

Common Architectural Patterns

Publish-Subscribe (Pub-Sub)

Concept: A messaging pattern where publishers send messages to topics, and subscribers receive messages from the topics they are interested in, without direct knowledge of each other.

Details:

  • Decouples publishers and subscribers, improving scalability and flexibility.
  • Facilitates asynchronous communication.

Important Questions:

  • How can we use the pub-sub pattern to notify multiple services of an event?
  • What are the challenges in ensuring message delivery and order?
  • What's the difference between a message queue and a pub-sub system?

CQRS (Command Query Responsibility Segregation)

Concept: A pattern that separates read and write operations for a data store.

Details:

  • Commands are tasks that change the state of the system.
  • Queries are tasks that read the state of the system but do not change it.
  • Allows for scaling the read and write models independently.

Important Questions:

  • When is it beneficial to use CQRS?
  • How do you keep the read and write models synchronized?
  • What are the consistency challenges with this pattern?

Circuit Breaker

Concept: A pattern to prevent a network or service failure from cascading to other services.

Details:

  • Acts as a proxy for operations that might fail. It monitors for failures and, after a certain threshold, "trips" or opens the circuit, causing subsequent calls to fail immediately without trying to execute the real operation.
  • After a timeout, it allows a limited number of test requests to pass through (half-open state) to check if the downstream service is healthy again.

Important Questions:

  • How can we make our system more resilient to failures in downstream services?
  • How does the circuit breaker pattern help prevent cascading failures?
  • What are the different states of a circuit breaker?
